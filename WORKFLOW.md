# WORKFLOW — 多智能体自动经济学研究工作流（统一版）

> **版本**：v3.0 (2026-03-26)
> **架构**：主研究 Agent + 独立 Critic Agent + 独立 Referee Agent + 红队 + 自动化审计脚本
> **目标**：从数据探索到最终论文，全流程自主推进
> **数据源**：OpenRouter 平台数据（日频面板 + 排行榜快照 + 模型元数据）

---

## 整体流程概览

```
Phase 0: 初始化
    ↓
Phase 1: 数据探索
    ↓
Phase 2: Idea 生成（3-5 个候选）
    ↓
Phase 3: Idea 数据验证（每个候选做快速数据检查）  ← 新增
    ↓
Phase 4: Idea 筛选与评审                          ← 新增
    ↓
  [GATE 0: Idea 可行性审查]
    ↓
Phase 5: 研究设计与预分析计划
    ↓
  [GATE 1: 选题与研究设计审查]
    ↓
Phase 6: 实证分析
    ↓
  [GATE 2: 方法论审查]
    ↓
Phase 7: 论文写作
    ↓
  [GATE 3: 论文终审]
    ↓
  [GATE 4: 修订审查]
    ↓
Phase 8: 模拟同行评审（最多 3 轮 R&R）
    ↓
  [最终决策: Accept / Reject]
```

---

## 核心设计原则

### 原则一：写作者和审查者分离
所有 Gate 和 Referee 由独立的 Agent 实例执行，不与主 Agent 共享对话上下文。

### 原则二：多元视角
同行评审阶段设置 5 类独立审稿人（领域、方法论、写作、政策、红队），每位在独立上下文中工作。

### 原则三：承诺机制
预分析计划在回归之前写定；每一轮修改生成独立版本（不直接改原稿）；全程 Git 版本控制。

### 原则四：对抗性检验
红队审稿人专注于攻击最薄弱假设，设计"杀手检验"。

### 原则五：数据先行，再选题（v3 新增）
不在纸面上评估 idea 的好坏，而是先让数据说话——每个候选 idea 都经过快速的数据探索验证，看数据特征是否支持该方向，再做最终选择。

## Phase 1：数据探索

**任务**：全面了解数据，为 idea 生成提供基础。

1. 编写 `code/01_data_exploration.py`
2. 加载数据源：
   - `data/master/usage_daily.csv` — 日频面板（模型×日期）
   - `data/rankings/` — 排行榜快照（5 天间隔，11 个维度）
   - `data/master/models_enriched_latest.csv` — 模型元数据
3. 产出：
   - 描述性统计表（面板维度、变量分布、缺失值）
   - ≥ 10 张探索性图表，覆盖不同角度：
     - 总量趋势和增长率
     - 市场集中度变化（HHI、CR5、CR10）
     - 厂商层面的竞争格局
     - 模型特征分布（价格、上下文长度、能力）
     - 进入/退出模式
     - 时间序列中的结构性变化
     - 任何有趣的异常或模式
   - `output/data_summary.md`：数据特征的完整描述

4. **刻意寻找意外**：不要只验证预期，主动寻找数据中"不应该存在"或"出乎意料"的模式。好的研究问题往往来自反常现象。

**硬指标门控**：≥ 10 张图 + 1 个完整描述性统计表 + data_summary.md → 通过

## Phase 2：Idea 生成

**任务**：基于数据探索结果，提出 3-5 个候选研究问题。

### 要求

每个候选 idea 必须包含：

1. **研究问题**：一句话描述
2. **经济学直觉**：为什么这个问题重要？对应什么理论框架？
3. **核心变量**：因变量、自变量、识别策略的初步构想
4. **数据可行性初判**：数据中有没有足够的变异来检验这个问题？
5. **与 paper_one 的差异**：不能与已完成论文重叠
6. **需要验证的关键假设**：列出 2-3 个"如果数据不支持，这个 idea 就不成立"的条件

### 选题方向提示

基于 OpenRouter 数据的特征，可以但不限于考虑：
- 定价策略与需求弹性（价格战、差异化定价、free tier 的作用）
- 开源 vs 闭源的竞争动态（市场份额、采用曲线、用户粘性）
- 推理能力 / 工具调用 / 多模态作为差异化维度
- 模型生命周期与"折旧"模式
- 平台层面的网络效应与锁定效应
- 市场信息结构（排行榜、benchmarks 对需求的影响）
- 模型组合使用行为（多归属 / routing 策略）

### ⚠️ 排除

- 不得重复 paper_one 的选题（需求再分配 / 创造性毁灭 / 进入效应）
- 必须有理论驱动，不接受纯描述性的数据报告
- 必须基于数据实际可行性，不要提出数据无法验证的问题

产出：`output/idea_candidates.md`

## Phase 3：Idea 数据验证（核心新增环节）

> **设计理念**：不在纸面上争论哪个 idea 更好，而是让数据来回答。每个候选 idea 进行 15-30 分钟的快速数据探索，验证关键假设是否被数据支持。

**任务**：对 Phase 2 的每个候选 idea，编写快速验证脚本并产出验证报告。

### 每个 Idea 的验证流程

1. **编写验证脚本** `code/02_idea_validation_{idea_name}.py`
   - 不需要完整的回归分析，只需要描述性证据
   - 目标：验证 Phase 2 中列出的"关键假设"

2. **核心检查项**（根据 idea 内容选择适用的）：

   **变异充分性**
   - 核心自变量的变异是否足够？（标准差、四分位距、跨组差异）
   - 如果做面板分析，组内变异是否足够？（固定效应吸收后还剩多少？）
   - 样本量在关键子样本中是否足够？

   **基本模式是否存在**
   - 画出核心关系的散点图 / binned scatter
   - 做一个最简单的双变量回归，看系数方向和显著性
   - 关键的描述性事实是否成立？（例如"开源模型比闭源便宜"——真的吗？差多少？）

   **识别策略可行性**
   - 如果打算用 DiD / Event Study：事件数量够吗？前趋势如何？
   - 如果打算用 IV：第一阶段 F 统计量？（哪怕是初步的）
   - 如果打算用 RDD：断点处的密度是否连续？是否有 manipulation？

   **潜在的致命问题**
   - 测量误差是否严重？核心变量有多"noisy"？
   - 是否有严重的选择偏误或存活偏误？
   - 时间跨度（93 天）是否足够捕捉这个现象？

3. **产出验证报告** `logs/idea_validation/idea_{N}_{name}.md`

   每份报告包括：
   ```
   # Idea {N}: {标题}

   ## 关键假设验证结果

   | 假设 | 验证方法 | 结果 | 判定 |
   |------|---------|------|------|
   | 假设 1 | ... | ... | ✅ 支持 / ⚠️ 存疑 / ❌ 不支持 |
   | 假设 2 | ... | ... | ... |
   | ... | ... | ... | ... |

   ## 数据特征摘要
   - 核心变量的描述性统计
   - 关键图表（保存到 output/idea_exploration/）

   ## 可行性评估
   - 数据支持度：高 / 中 / 低
   - 主要风险：...
   - 预期效应量级：...（基于简单回归的粗略估计）

   ## 总结建议
   - 推荐 / 有保留推荐 / 不推荐
   - 如果推荐，下一步最需要做什么
   - 如果不推荐，原因是什么
   ```

### ⚠️ 行为规范

- **每个 idea 的验证是独立的**：不要因为 idea 1 看起来不错就敷衍 idea 2 的验证
- **诚实报告**：如果数据不支持，就说不支持。不要为了"凑够好的 idea"而美化结果
- **快速但不草率**：目标是 15-30 分钟的探索，不是完整分析。但图要看仔细，数字要核实
- **记录意外发现**：验证过程中看到的有趣模式，即使和当前 idea 无关，也记录下来

## Phase 4：Idea 筛选与评审

**任务**：基于数据验证结果，选出最优 idea。

### 筛选流程

1. **汇总对比表**：把所有 idea 的验证结果放在一起比较

   ```
   | Idea | 数据支持度 | 识别策略可行性 | 预期贡献 | 主要风险 | 综合排序 |
   |------|-----------|--------------|---------|---------|---------|
   | 1 | ... | ... | ... | ... | ... |
   | 2 | ... | ... | ... | ... | ... |
   ```

2. **选择标准**（按优先级排序）：
   - 数据是否支持？（一票否决：数据不支持的直接淘汰）
   - 识别策略是否可信？（从 paper_one 吸取教训：没有可信 IV 不要做因果声明）
   - 边际贡献是否清晰？
   - 时间跨度和样本量是否足够？
   - 研究问题是否有趣且重要？

3. **写选题决策书** `output/idea_selection.md`：
   - 选了哪个，为什么
   - 没选的各自为什么被淘汰
   - 被选 idea 的主要风险和应对预案
   - 如果被选 idea 在后续阶段遇到致命问题，备选方案是什么

### 🚧 GATE 0：Idea 可行性审查（新增）

**独立 Agent 评分**（4 项 × 1-10 分）：

| 维度 | 评估内容 | 权重 |
|------|---------|------|
| 数据支持度 | 验证报告中的关键假设是否被数据支持？ | 30% |
| 识别策略可行性 | 基于数据特征，识别策略能走通吗？ | 30% |
| 边际贡献 | 相对于已有文献（含 paper_one），贡献是否清晰？ | 20% |
| 研究问题重要性 | 经济学意义上是否值得研究？ | 20% |

**通过标准**：加权平均 ≥ 6.5

**不通过处理**：
- 如果是数据支持度或识别策略不足 → 换候选 idea（回到筛选）
- 如果是贡献或重要性不足 → 修改定位后重审
- 最多 2 轮；2 轮不过 → 重新回到 Phase 2 生成新的 idea

**Critic 特别关注**：
- 验证报告中是否有被忽视的红旗（red flags）？
- paper_one 遇到的 IV 失败问题，在新 idea 中是否有应对？
- 93 天面板长度对新 idea 是否构成约束？

## Phase 5：研究设计与预分析计划

**前提**：通过 Gate 0

### 5a. 完整研究设计

1. 基于已选定的 idea，写完整的研究设计 `output/research_design.md`：
   - 完整理论框架（经济主体、目标函数、均衡预测、≥ 3 个可检验假说）
   - Identification Memo（`output/identification_memo.md`）：
     - 回归方程中的因果机制
     - 核心识别假设
     - ≥ 3 个最大识别威胁 + 每个的应对方案
     - **如果做不到可信的因果识别 → 论文必须明确定位为描述性研究**（paper_one 的教训）
   - 最接近的 3 篇已有论文 + 逐篇边际贡献说明

2. **文献搜索**：
   - 搜索 NBER Working Papers、Google Scholar、SSRN
   - 重点搜索与选定 idea 直接相关的文献
   - 所有引用上网验证真实性
   - 记录到 `logs/literature_log.md`

### 5b. 预分析计划

编写 `output/pre_analysis_plan.md`：
- 主回归方程（完整写出）
- 变量定义（因变量、核心自变量、控制变量、固定效应）
- 聚类标准误层级 + 理由
- 预期系数方向 + 经济学理由
- 稳健性检验计划（≥ 5 种）
- 异质性分析计划（≥ 3 个维度）

### 🚧 GATE 1：选题与研究设计审查

独立 Agent 评分（5 项 × 1-10 分）：
- 经济学意义
- 数据匹配度
- 识别可行性（**重点：IV 或类实验设计的可信度**）
- 边际贡献
- 可行性

**≥ 6.5 通过** → 进入 Phase 6
**< 6.5** → 修改后重审，最多 3 轮

## Phase 6：实证分析

### 6a. 数据构建
1. 编写 `code/03_data_construction.py`
2. 变量构造、样本筛选、面板结构处理
3. 输出 `output/analysis_panel.csv`

1. 编写 `code/04_baseline_regression.py`

5. **经济显著性强制报告**：
   - 每个核心系数：1-SD 变化 → Y 变化（绝对值 + 百分比）

### 6c. 稳健性检验
1. 编写 `code/05_robustness.py`
2. 按预分析计划执行 ≥ 5 种检验
3. 所有结果标准格式保存

### 6d. 异质性分析
1. 编写 `code/06_heterogeneity.py`
2. ≥ 3 个维度
3. 交互项回归 + 分组回归

### 6e. 文献搜索（第二轮）
- 回归结果出来后搜索类似实证发现
- 对比效应大小和方向
- 补充到 `logs/literature_log.md`

### 防线规则
- 每次 specification 变更记录在 `logs/research_log.md`
- 基准结果不显著 → 诚实报告，不 p-hack
- 与预分析计划的偏离必须解释理由
- 迭代 specification 最多 5 轮

### 🚧 GATE 2：方法论审查

独立 Agent 评分（5 项 × 1-10 分）：
- 模型设定
- 标准误处理
- 识别策略（含 pre_analysis_plan 偏离检查）
- 稳健性检验覆盖
- 结果解读（经济显著性 + 因果/描述性语言的准确性）

**≥ 7.0 通过** → 进入 Phase 7
**< 7.0** → 修改后重审，最多 3 轮

## Phase 7：论文写作

**输出格式**：LaTeX（可编译为 PDF）

### ⚠️ 长文档分步生成规则

**禁止一次性生成整篇论文。** LaTeX 论文必须按章节分步写作，每次只生成一个 section：

1. 先创建 `paper/main.tex` 框架
2. 按顺序逐节填充：
   - Title + Abstract（≤ 200 词）
   - Introduction（≤ 2500 词）
   - Institutional Background / Data
   - Theoretical Framework / Model
   - Empirical Strategy
   - Results
   - Robustness + Heterogeneity
   - Discussion + Conclusion
   - References（用 BibTeX）
3. 每节写完后编译检查

**理由**：一次性生成 10,000+ 字的 LaTeX 会超出单次输出限制，导致截断和前后不一致。同理，任何超过 3000 字的文档都应分段写入。

### 写作标准
- ≥ 10,000 字（不含参考文献和附录）
- ≥ 20 篇引用，全部上网验证
- 0 个伪造引用
- 三线表 + 专业图表

### 数字验真

论文写完后、Gate 3 之前运行 `code/07_verify_numbers.py`：
- 遍历论文中所有数字，与 `output/tables/` 逐一比对
- 输出 `logs/number_verification.md`
- 任何不匹配必须修正

### 代码复现检查

最终打包前从头重跑全部代码，确保可复现。

### 🚧 GATE 3：论文终审

独立 Agent 评分（5 项 × 1-10 分）：
- 论点清晰度
- 实证扎实度
- 写作质量
- 学术诚实度
- 发表可能性

**≥ 7.0 通过** → 进入 Gate 4
**< 7.0** → 修改后重审，最多 3 轮

### 🚧 GATE 4：修订审查

独立 Agent 专注于：
- 分析补充（遗漏的稳健性、异质性）
- 写作质量提升
- 格式和一致性

**≥ 7.5 通过** → 进入 Phase 8
**< 7.5** → 修改后重审，最多 3 轮

## Phase 8：模拟同行评审

### 审稿团队

5 类审稿人，每位由完全独立的 Agent 实例担任：

| 角色 | 关注重点 |
|------|---------|
| Referee 1：领域专家 | 文献覆盖、制度细节、研究问题重要性 |
| Referee 2：方法论专家 | 因果识别、计量模型、统计推断 |
| Referee 3：写作专家 | 叙事结构、语言质量、AI 写作痕迹 |
| Referee 4：政策专家 | 政策含义、外部有效性、现实相关性 |
| 红队审稿人 | 攻击最薄弱假设，设计杀手检验 |

### 评审标准

对标顶级经济学期刊（AER / AEJ / QJE / REStud）。

每位审稿人独立评分（6 项 × 1-10 分）：
- 文献定位与边际贡献
- 方法论严谨性
- 结果可信度与经济解读
- 写作与呈现质量
- 研究问题重要性
- 总体发表准备度

### 评审流程

**R1（全部审稿人参与）→ 意见分类 → 修改 → R2 → ... → 最多 R3**

1. **收到意见后先分类**：
   - Must Fix：多数审稿人一致指出的核心问题
   - Should Fix：有道理但非核心的建议
   - Consider：个别审稿人的偏好
   - Decline：互相矛盾的意见，礼貌拒绝并解释

2. **撰写 Response Letter**：逐条回应每位审稿人

3. **修改论文**：生成新版本（main_r1.tex, main_r2.tex），修改部分蓝色标注

4. **动态参与**：
   - 给了 Accept 的审稿人退出后续轮次
   - 未 Accept 的继续参与
   - 红队在平均分 < 6.5 时自动启动

5. **终止条件**：
   - 所有审稿人在某轮给出 Accept → 流程结束
   - 3 轮用完 → 以最终状态结束
   - 任何轮次平均分 < 4.0 → 建议放弃该论文

## ⚠️ 行为规范

### 长文档分步生成
禁止一次性生成超过 3000 字的文档。LaTeX 论文按章节写，长 markdown 分段输出。

### 诚实性红线
- 基准结果不显著 → 诚实报告，不 p-hack
- 识别策略有缺陷 → 承认局限性，不伪装因果
- 引用不确定 → 上网搜索验证，不瞎编
- 数字不匹配 → 修正，不掩盖

### 版本控制
- 每个关键阶段完成后 git commit
- Commit message 格式：`{phase}: {描述}`
- 论文修改生成新版本文件，不直接修改原稿
- 保留所有中间产出，方便横向对比

### 与已有论文的关系
- 阅读 `paper_one/` 的研究设计和结果，了解已做的工作
- 避免选题重叠
- 可以引用 paper_one 的发现作为动机或对照
- 共享数据源，但独立编写代码（可参考但不复制）

## Critic Agent 执行规范

### 通用规则
- Critic 是**独立的 Agent 实例**，不与主 Agent 共享对话上下文
- Critic 必须给出**具体的、可操作的修改建议**，不能只说"需要改进"
- 每项评分必须附带理由（≥ 2 句话）
- Critic 必须检查与预分析计划的一致性

### Gate 0 特别规则（新增）
- 重点审查**数据验证报告的可信度**：验证过程是否充分？有没有被忽视的 red flags？
- 评估 paper_one 遇到的问题（IV 失败、93 天面板限制）在新 idea 中是否有应对
- 如果所有候选 idea 的数据支持度都不够 → 建议重新探索数据、寻找新方向

### 审稿人特别规则
- 每位审稿人在**独立上下文窗口**中工作
- 红队审稿人的杀手检验必须是**具体的、可执行的**统计检验
- 审稿人必须阅读完整论文 + 相关数据摘要才能出具报告
